# 游戏化室内巡检系统 - 开发执行计划

> 项目名称：AI Patrol Bot（室内巡逻机器人）
> 创建日期：2026-07-01
> 项目位置：`game/`（与现有 `demo/`、`src/` 隔离，互不影响）

---

## 一、项目定位

将现有的 AI 视觉室内巡检系统包装成星露谷物语风格的 2D 俯视角游戏。玩家控制一个巡逻机器人在室内移动，到达风险区域时触发 VLM 识别，以游戏化方式完成室内安全巡检。

### 核心体验

1. 玩家用 WASD 或方向键控制机器人在室内走动
2. 机器人在房间中巡逻，经过风险高发区域时触发提醒
3. 游戏截图当前画面 → 调用现有 `/api/inspect` 接口 → 展示 VLM 识别结果
4. 室内场景（家具、风险点位置）可通过 JSON 配置自定义

### 与现有系统的关系

```
game/                    # 新项目（Phaser 游戏前端）
├── index.html           # 游戏入口
├── js/                  # 游戏逻辑
├── assets/              # 精灵图、瓦片图
├── scenes/              # 场景配置 JSON
└── DEVELOPMENT_PLAN.md  # 本文件

src/server.py            # 现有后端（不改动，直接复用 /api/inspect）
configs/risk_rules.json  # 现有风险规则（直接复用）
demo/                    # 现有 demo（不改动）
```

**原则：只读复用现有后端 API，不修改 demo/ 和 src/ 的任何代码。**

## 二、技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 游戏引擎 | Phaser.js 3.80 | 最成熟的 Web 2D 游戏框架，内置物理引擎、Tilemap、输入系统 |
| 地图编辑 | Tiled 1.10 | 开源 Tilemap 编辑器，导出 JSON 格式，Phaser 原生支持 |
| 美术素材 | OpenGameArt + 自制 SVG | 免费 2D 像素素材，机器人用简单 SVG 绘制 |
| 物理引擎 | Arcade Physics | Phaser 内置，轻量 AABB 碰撞检测够用 |
| 前端构建 | 无构建工具 | 直接 `<script>` 引入 Phaser CDN，零配置 |
| 后端 | 现有 FastAPI | 复用 `/api/inspect`、`/api/providers`、`/api/risk_rules` |

## 六、VLM 集成方案

### 架构设计：导航地图 + 相机面板

游戏画面（俯视 2D 像素风）仅负责巡逻导航，不作为 VLM 分析目标。
机器人进入风险区域后，弹出"摄像头面板"展示预生成的真实室内照片，
将该照片发送给 VLM 进行检测。

```
游戏画布（导航地图）              相机面板（VLM 分析目标）
┌──────────────────┐            ┌───────────────────────┐
│  [床]  [书桌]    │            │  [真实室内照片]        │
│        🤖→       │  进入      │  水杯紧贴墙角插座      │
│  [风险区域]      │ ────────>  │                       │
│  [垃圾桶]        │  风险区    │  → 发送给 VLM         │
└──────────────────┘            └───────────────────────┘
```

### 图片来源：预生成图片库

| 来源 | 说明 | VLM 检测价值 | 成本 |
|------|------|-------------|------|
| AI 生成真实室内照片 | 开发阶段一次性生成，存为本地 PNG | 高 | 一次性，约 20-30 张 |
| 用户上传真实照片 | 场景编辑器中为风险区域上传照片 | 最高 | 零 API 成本 |
| 现有 demo SVG 转 PNG | 从 `demo/assets/scenes/` 转换 | 中低 | 零成本 |

游戏运行时只加载本地静态文件，不调用图像生成 API。

图片库按风险类型分类，场景编辑器中按风险类型自动匹配：

```
game/assets/camera_views/
  ├── electric_shock_01.png    ← 水杯靠近插座
  ├── electric_shock_02.png    ← 积水靠近电源线
  ├── fire_01.png              ← 易燃物靠近灶台
  ├── trip_01.png              ← 纸箱阻塞通道
  ├── trip_02.png              ← 电线横跨地面
  ├── cleanliness_01.png       ← 垃圾堆积
  ├── chemical_01.png          ← 化学品暴露
  └── ...                      ← 每类风险 2-3 张
```

### 调用流程

```
机器人进入风险区域
  → 弹出相机面板，展示该区域关联的预生成照片
  → 将照片文件发送到 /api/inspect
  → 等待 VLM 返回结果
  → 在相机面板中叠加风险识别结果
  → 记录到巡逻日志
```

### Provider 选择

游戏顶栏新增设置按钮，弹出面板可选择：
- `mock`：无网络无 Key，返回预设结果（课堂演示用）
- `qwen`：Qwen-VL 真实调用（需输入 DASHSCOPE_API_KEY）
- `openai`：OpenAI GPT-4V 真实调用（需输入 OPENAI_API_KEY）
- `ollama`：本地视觉模型（无需 Key）

API Key 仅保存在当前会话内存中，不写入磁盘。

### 游戏专用 Prompt

创建 `configs/prompts/game_inspection_v1.md`，与真实照片 Prompt 的差异：
- 声明输入为室内场景照片（非游戏截图）
- 传入风险区域的 `prompt_hint` 作为巡检重点提示
- 要求模型说明从照片中观察到的物体和空间关系

### 关键实现

- 相机面板：HTML overlay div，展示预生成图片 + VLM 结果
- 图片加载：Phaser `this.load.image()` 预加载或 `fetch` 按需加载
- 调用：与 `inspect.js` 相同的 fetch 逻辑，provider 由用户选择
- 降级：调用失败时展示错误信息，不影响游戏继续

## 八、风险与应对

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| Phaser 学习曲线 | 中 | 前期慢 | 官方示例丰富，半天可上手基础 |
| 素材风格不统一 | 中 | 视觉差 | 统一用 16x16 像素风格，或全部自制 SVG |
| VLM 调用延迟 | 高 | 游戏卡顿 | 异步调用，先弹"检测中"动画，不阻塞游戏 |
| Canvas 截图质量 | 低 | 识别率 | 确保截图分辨率 ≥ 512x512 |
| 场景编辑器复杂 | 中 | 延期 | G3 阶段可独立跳过，不影响 G1/G2 |
| 碰撞检测卡墙 | 中 | 体验差 | 用 Arcade Physics 的 tilemap 碰撞，成熟方案 |

## 十、开发顺序建议

```
G1 (MVP)  ──→  确认效果  ──→  G2 (多房间)  ──→  G3 (编辑器)  ──→  G4 (游戏化)
   3天           0.5天           2天              2天              2天
```

**总计约 9-10 天**，每个阶段可独立交付，随时可停。

建议先完成 G1 看到效果后再决定是否继续后续阶段。
