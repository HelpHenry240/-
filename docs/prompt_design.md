# Prompt 设计文档

本文档记录 AI 视觉室内巡检系统中三版巡检 Prompt 的设计思路、版本演进、结构化输出 schema 以及约束策略。Prompt 文件统一存放于 `configs/prompts/` 目录，配合 `configs/risk_rules.json` 风险规则文件使用。

---

## 1. 设计目标

巡检 Prompt 需要同时满足以下要求：

1. **可控性**：模型输出必须遵循固定 JSON schema，便于前端展示和自动评测。
2. **准确性**：风险判断必须基于空间关系，而非物体本身，避免过度报警。
3. **可解释性**：每条风险必须附带 reason 和 suggestion，便于人工核查。
4. **不确定性表达**：当视角、遮挡或分辨率不足以判断时，模型应主动报告而非强行判定。
5. **可对比性**：不同版本 Prompt 可在同一数据集上批量运行并横向对比指标。

---

## 2. Prompt 版本说明

系统设计了三个递进式 Prompt 版本，对应文件如下：

| 版本 | 文件路径 | 定位 |
|------|---------|------|
| v1 基线版 | `configs/prompts/risk_inspection_v1.md` | 最小可用基线 |
| v2 细化规则版 | `configs/prompts/risk_inspection_v2.md` | 增加距离阈值与排除条件 |
| v3 带正负示例版 | `configs/prompts/risk_inspection_v3.md` | 增加 few-shot 正负示例 |

### 2.1 v1 基线版

**设计理由**：

v1 版本是课程最小闭环的起点，目标是验证「VLM 能否根据自然语言规则输出结构化 JSON」。该版本仅给出 6 条风险规则的文字描述和输出 JSON 模板，不包含距离阈值、排除条件或示例。

**核心内容**：

- 定义 5 类关注区域（地面通道、台面边缘、插座电器、儿童可触达位置、垃圾区域）。
- 列出 6 条风险规则，每条以一句话描述触发条件。
- 给出输出 JSON 模板，包含 `has_risk`、`risks`、`evidence_sufficiency`、`uncertain_points` 四个顶层字段。
- 约束：要求说明空间关系，视角不足时写入 `uncertain_points`。

**已知不足**：

- 规则描述过于宽泛，未给出距离阈值，容易导致误检（如 S05 正常厨房被误判为火灾风险）。
- 等级标准模糊，仅用「低/中/高」描述，未给出各级别定义（如 S07 化学品暴露等级被低估）。
- 缺少正负示例，模型对「正常摆放」与「风险摆放」的边界理解不足。

### 2.2 v2 细化规则版

**设计理由**：

v1 评测暴露的主要问题是误检和等级错误。v2 的改进方向是为每条规则增加可量化的触发条件、排除条件和等级标准，让模型有明确的判断依据而非主观猜测。

**改进点**：

- **引入距离阈值**：例如触电风险要求液体与插座距离小于 20cm 才触发，火灾风险要求易燃物与热源距离小于 30cm。
- **增加排除条件**：例如液体在桌面中央且距插座超过 30cm 时不触发，热源已关闭且冷却时不触发。这直接针对 S05 误检问题。
- **明确等级标准**：每个风险类型给出「高/中/低」的具体定义。例如危险物暴露规则中，「无收纳且靠近活动区域为高」「有防护但位置偏低为中」。
- **强化核心原则**：在 Prompt 开头增加三条核心原则——不过度报警、空间关系优先、证据不足时报告。
- **细化关注区域描述**：为每类区域补充判断维度，如「通行宽度是否被压缩」「高处物品是否稳定」。

**预期效果**：

- 降低误检率：排除条件让正常场景不再被触发。
- 降低等级错误率：明确的等级标准让模型判定更一致。

### 2.3 v3 带正负示例版

**设计理由**：

v2 通过规则细化解决了「规则不清」的问题，但模型仍可能因为缺少参照而对边界场景判断不准。v3 引入 few-shot 正负示例，让模型通过对比学习「什么该判、什么不该判」。

**改进点**：

- **正例（应判为风险）**：给出 3 个典型正例，覆盖触电、火灾、通行三类风险，每个正例包含场景描述、判定结果和推理理由。
- **负例（不应判为风险）**：给出 3 个典型负例，对应正常摆放、正常厨房、整齐收纳，说明为何不触发规则。负例直接对应 S05 误检场景。
- **等级判断标准表**：以表格形式给出「高/中/低」的定义和典型场景，例如「高=直接危险需立即处理」「中=潜在危险需尽快处理」「低=轻微隐患可择期处理」。
- **参照推理约束**：要求模型参照正例和负例的推理方式判断，而非自行发挥。

**预期效果**：

- 进一步降低误检：负例让模型明确「正常场景应返回 has_risk: false」。
- 提升推理一致性：正例提供了标准化的推理范式。

## 3. 结构化输出 JSON Schema

三个版本共享同一套输出 schema，确保前端和评测逻辑无需随 Prompt 版本变化。

### 3.1 顶层结构

```json
{
  "has_risk": true,
  "risks": [RiskItem],
  "evidence_sufficiency": "充分 | 不充分",
  "uncertain_points": ["string"]
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `has_risk` | boolean | 是否存在风险。正常场景为 false，risks 为空数组。 |
| `risks` | array[RiskItem] | 检出的风险列表，每项包含类型、物体、位置、等级、理由、建议。 |
| `evidence_sufficiency` | string | 证据充分性判断。当视角或遮挡不足时为「不充分」。 |
| `uncertain_points` | array[string] | 不确定点说明，用于报告遮挡、视角不足等问题。 |

### 3.2 RiskItem 结构

```json
{
  "type": "触电风险 | 火灾风险 | 通行/绊倒风险 | 坠物风险 | 危险物暴露 | 清洁度异常",
  "objects": ["相关物体1", "相关物体2"],
  "location": "图像中的位置描述",
  "level": "低 | 中 | 高",
  "reason": "引用触发规则并说明图像证据",
  "suggestion": "处理建议"
}
```

### 3.3 Schema 设计原则

1. **扁平化**：避免深层嵌套，前端渲染和评测匹配都更简单。
2. **枚举约束**：`type` 和 `level` 使用固定枚举值，便于精确匹配（评测时做字符串归一化）。
3. **证据与判断分离**：`reason` 说明判断依据，`evidence_sufficiency` 和 `uncertain_points` 说明证据质量，二者解耦。
4. **可扩展**：后续可增加 `risk_type_code`（英文编码）字段用于跨语言对比，不影响现有字段。

### 3.4 后端校验

`src/datasets/schema.py` 中的 `validate_model_output` 函数对模型输出做字段校验：

- 必须是 JSON 对象。
- 必须包含 `has_risk` 和 `risks` 字段。
- `risks` 必须是数组，每项必须包含 `type` 和 `level`。
- 校验失败时保留错误信息但不丢弃结果，便于排查格式问题。

## 4. 约束策略

### 4.1 防止过度报警

过度报警（误检）是巡检系统的主要痛点。三个版本采用递进式约束：

| 策略 | v1 | v2 | v3 |
|------|----|----|-----|
| 要求说明空间关系 | 有 | 有 | 有 |
| 距离阈值 | 无 | 有 | 有 |
| 排除条件 | 无 | 有 | 有 |
| 负例参照 | 无 | 无 | 有 |
| 正常场景返回 false | 提及 | 强调 | 强调 |

### 4.2 证据充分性判断

`evidence_sufficiency` 和 `uncertain_points` 字段的设计目的是让模型主动报告「看不清」的情况，而非强行判定：

- 当视角被遮挡、物体过小、或距离无法确认时，模型应将 `evidence_sufficiency` 设为「不充分」。
- `uncertain_points` 用于具体描述哪些区域或物体无法确认。
- 这两个字段在评测中不直接参与对错判定，但作为失败案例归因的重要依据。例如 S12 漏检案例中，如果模型能报告「桌下区域被遮挡，无法确认」，则可以提示补充视角而非判定为完全失败。

### 4.3 等级标准

v2 和 v3 为每个风险类型定义了等级标准，核心原则是「等级与危险程度一致」：

- **高**：直接危险，需立即处理。如液体直接接触插座、易燃物直接接触热源、出口被堵。
- **中**：潜在危险，需尽快处理。如液体靠近电器（10-30cm）、高处边缘物品、通道部分受阻。
- **低**：轻微隐患，可择期处理。如少量垃圾堆积、物品摆放略有不当。

v3 进一步用表格形式固化等级定义，减少模型对等级的主观偏差。

### 4.4 输出格式约束

所有版本都要求「只输出 JSON，不要输出 Markdown 代码块或额外解释」。后端 `QwenProvider` 的 `_parse_json` 函数做了鲁棒解析：

1. 先尝试直接 `json.loads`。
2. 失败则尝试提取 ` ```json ... ``` ` 代码块。
3. 再失败则尝试提取第一个 `{` 到最后一个 `}` 之间的片段。
4. 全部失败则保留原始文本并标记解析错误。

这种多级回退策略保证了即使模型未严格遵循格式约束，也能尽可能解析出有效结果。

## 5. 风险规则与 Prompt 的关系

风险规则的唯一来源是 `configs/risk_rules.json`，每条规则包含：

- `rule_id`：规则唯一标识（如 `electric_shock_001`）。
- `risk_type` / `risk_name`：英文编码与中文名称。
- `description`：规则描述。
- `trigger_conditions`：触发条件列表。
- `level_policy`：等级判定策略。
- `positive_examples` / `negative_examples`：正负示例。

`src/prompts/builder.py` 中的 `PromptBuilder` 负责加载规则文件和 Prompt 模板。当前三个版本的 Prompt 文件中已内联了规则文本，后续可通过 `PromptBuilder` 从 `risk_rules.json` 动态生成 Prompt，实现规则与 Prompt 的单一数据源。

## 6. 多 Prompt 对比机制

系统支持对同一数据集用不同 Prompt 批量运行并自动对比：

1. **批量运行**：`POST /api/run_multi_batch` 接口接收逗号分隔的 `prompt_ids`，逐个 Prompt 运行全量数据集。
2. **自动评测**：每次运行后自动调用 `evaluate_run` 计算准确率、漏检数、误检数、等级错误数等指标。
3. **横向对比**：`compare_runs` 函数构建逐样本对比表，找出不同 Prompt 结果不一致的「差异样本」。
4. **最佳 Prompt 选取**：对比摘要自动标注准确率最高的 run，辅助判断哪个 Prompt 版本效果最好。

已有对比数据：`20260630_180028_mock_risk_inspection_v1` 与 `20260630_180028_mock_risk_inspection_v2` 可通过 `/api/compare` 接口对比。

---

## 7. 总结

三版 Prompt 的演进逻辑是「先跑通闭环，再提升精度」：

- v1 验证可行性：VLM 能输出结构化 JSON。
- v2 解决规则模糊：引入距离阈值和等级标准。
- v3 解决边界判断：引入正负示例和等级表。

每一版的改进都直接对应 mock 评测中暴露的失败案例，体现了「评测驱动 Prompt 优化」的闭环思路。
