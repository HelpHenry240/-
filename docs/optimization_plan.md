# 后续优化方案

> 基于「AI 视觉室内巡检系统」当前实现（阶段 A-D 已完成）的后续优化方向。
> 更新日期：2026-06-30

---

## 一、API 接口泛化

| 方向 | 当前状态 | 优化建议 |
|------|---------|---------|
| 多图输入 | 仅支持单图上传 | `/api/inspect` 改为 `List[UploadFile]`，支持多视角图片同时传入，对应 schema 中的 `media` 数组 |
| 自定义 endpoint | 4 种固定 provider | 增加 `custom_http` provider，允许用户在前端填写 base_url、headers、payload 模板，适配任意 OpenAI 兼容服务 |
| 流式输出 | 一次性返回全部结果 | 接入 SSE（Server-Sent Events），模型生成时逐步推送文本，前端实时展示推理过程 |
| 批量异步 | `run_batch` 是同步阻塞的 | 改为后台任务（BackgroundTasks 或 Celery），返回 task_id，前端轮询进度 |
| API 鉴权 | 无鉴权 | 加 API Token 或 session 校验，避免局域网内被随意调用 |
| Webhook 回调 | 无 | 批量运行完成后可回调通知前端，而非前端轮询 |

## 二、前端界面优化

### 2.1 交互体验

- 实时巡检页增加**拖拽调整图片**、旋转、裁剪功能，方便用户调整输入
- 结果展示增加**风险热区标注**——在图片上用边框高亮风险区域，比纯文字描述更直观
- 批量评测页增加**筛选器**：按风险类型、场景类型、错误类型、模型名称筛选样本
- 对比页增加**柱状图/雷达图**可视化指标差异，目前只有表格

### 2.2 视觉设计

- 统一设计系统：当前 `styles.css` 是 demo 用的，新页面用了内联样式，应该抽成共享组件库
- 响应式适配：当前在移动端布局会挤压，需要 media query 优化
- 深色模式：增加 `prefers-color-scheme` 支持

### 2.3 功能增强

- 历史结果对比：在批量评测页选择两个历史 run 直接对比，不用重新跑
- 结果导出：一键导出评测报告为 PDF 或 HTML
- 图片标注工具：上传真实照片后可在前端画框标注 ground truth，降低标注成本

## 三、数据层优化

- **标注工具**：目前 ground truth 是手写 JSON，应做一个可视化标注界面——在图片上框选区域、选择风险类型、填写等级，自动生成标注 JSON
- **数据增强**：对真实照片做翻转、亮度调整、裁剪等增强，扩充样本量
- **数据集版本管理**：每次导入文件夹时记录版本和哈希，避免数据变更后历史 run 不可复现
- **公开数据集接入**：项目里已有 HomeSafeBench、BEHAVIOR-1K、ESI-BENCH 三个 PDF，可以写转换脚本将这些数据集统一转为项目 schema

---

## 四、模型层优化

- **多视角融合**：同一场景拍 2-3 张不同角度的照片，一起传给模型，对比单视角的识别效果差异
- **短视频抽帧**：上传短视频，后端用 ffmpeg 抽帧后逐帧或选关键帧调用 VLM
- **检测 baseline**：在 VLM 之前先用 YOLO 等目标检测模型定位物体，再把检测框坐标传给 VLM 做关系推理，降低漏检率
- **成本估算**：记录每次调用的 token 数和费用，在 run 记录中增加 `cost_estimate` 字段
- **重试与降级**：API 调用失败时自动重试 1-2 次，仍失败则降级为 mock 并标注

---

## 五、评测层优化

- **更多指标**：增加精确率（Precision）、召回率（Recall）、F1 分数，目前只有准确率
- **混淆矩阵**：按风险类型生成混淆矩阵，看哪些类型容易混淆
- **一致性分析**：同一 Prompt 同一图片跑 N 次，统计输出稳定性（temperature > 0 时结果可能不同）
- **自动报告生成**：`src/reports/render.py` 目前是占位，应该实现自动生成 HTML 巡检报告（含图表、失败案例、改进建议）

---

## 六、工程化优化

- **测试覆盖**：`tests/` 目前为空，应该补单元测试（schema 校验、评测逻辑、provider mock）和集成测试
- **CI/CD**：加 GitHub Actions，每次提交自动跑 `python -m pytest` 和语法检查
- **Docker 化**：写 Dockerfile，一键启动后端 + 前端，避免环境配置问题
- **日志系统**：用 logging 替代 print，记录 API 调用、错误、耗时
- **配置校验**：启动时校验 `configs/providers.json` 格式，提前发现配置错误

## 七、课程交付优化

- **最终报告**：`docs/final_report.md` 还未创建，应整合评测数据、失败案例、Prompt 对比结论
- **展示 PPT**：项目里已有 `docs/AI室内巡检.pptx`，可以更新为最新系统截图和数据
- **演示视频**：录制一段完整流程演示（上传图片 → Qwen-VL 识别 → 批量评测 → Prompt 对比）
- **Live Demo 数据**：预置几组真实照片的 run 结果，演示时即使网络不通也能展示真实模型输出

## 八、优先级排序

建议按以下顺序推进：

| 优先级 | 方向 | 理由 |
|--------|------|------|
| P0 | 标注工具 | 降低数据准备成本，影响最大 |
| P0 | 多图输入 + 多视角融合 | 课程评价重点，直接提升识别能力 |
| P1 | 前端风险热区标注 | 演示效果提升明显 |
| P1 | 检测 baseline | 降低漏检率，技术深度高 |
| P1 | 自动报告生成 | 课程交付完整性 |
| P2 | 评测指标扩展（Precision/Recall/F1） | 评测更专业 |
| P2 | 测试 + Docker | 工程化收尾 |
| P2 | 最终报告 + PPT 更新 | 课程交付收尾 |
| P3 | 流式输出、鉴权、CI/CD | 长期维护方向 |
